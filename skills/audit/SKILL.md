# Vibe Check — Production Readiness Audit

The entry point. AI tools get you to something that *runs* surprisingly fast, but "runs on my screen"
and "survives real users" are different bars. This audit finds the gap — without pretending every app
needs everything.

**This is adaptive, not a fixed checklist.** A static landing page is not a multi-tenant SaaS. The
audit figures out what your app *is* first, then only checks what's relevant and marks the rest as
*not applicable*. Routing, not lecturing.

## When to Use This Skill

- User wants to "ship", "launch", "go live", or asks "is this production-ready?".
- User asks "what am I missing before launch?".
- A working demo "falls apart when real people use it".
- User is hardening an app generated by Lovable, Cursor, Bolt, v0, Replit, etc.
- User says "it works, but…" (slow, insecure, crashes, no logs, scary bill).

## How It Works

### Step 1 — Profile the app (don't assume)

Infer from the code, or ask the few questions you can't answer. The profile decides which areas apply:

- **Does it have a backend/server**, or is it purely static/client-side?
- **Does it use a database?** Which one?
- **Are there user accounts / login?** Is it single-user or multi-tenant (B2B)?
- **Does it call an LLM or paid AI API?**
- **Does it take payments / charge money?**
- **Is there a mobile or native surface**, or web only?
- **Does it collect personal data?** Any enterprise or app-store distribution planned?
- **What traffic is realistic** at launch and 3–6 months out?

### Step 2 — Select the areas that apply

Use the matrix to include or skip each area. Skipping is a real answer — don't flag what doesn't exist.

| Area | Applies when… | Route to |
|---|---|---|
| Code ownership & launch gaps | Always (any AI-built app) | `production-readiness` |
| Security & secrets | Always — any app with code/keys | `app-security` |
| Auth & access control | There are user accounts / protected data | `auth-access` |
| Data & schema | There's a database | `data-architecture` |
| Scaling & performance | There's a backend/DB **and** real traffic is expected | `scaling-performance` |
| AI / LLM engineering | The app calls an LLM / paid AI API | `ai-engineering` |
| Observability | There's a backend (anything that can fail server-side) | `observability` |
| Deployment & CI/CD | Anything you redeploy (almost all; depth scales with team) | `deployment-cicd` |
| Reliability & recovery | You hold user data or promise uptime | `reliability-recovery` |
| Compliance & legal | You collect personal data, sell B2B, or ship to an app store | `compliance-legal` |
| API & backend boundary | There's a backend/API the client calls | `api-architecture` |
| Frontend & mobile quality | There's a UI, especially mobile | `frontend-mobile-quality` |
| Cost & infrastructure | Infra/usage is non-trivial enough to matter | `cost-infrastructure` |
| Monetization & billing | The app charges money | `monetization-pricing` |

### Step 3 — Triage by risk, for *this* app

Rank the applicable gaps by damage-if-ignored, not by list order. Things that leak data, let one user
read another's, lose data, or fall over under load come first. Cosmetic and nice-to-have come last.

### Step 4 — Route and report

For each applicable area, mark `solid` / `weak` / `missing` / `n/a`, then hand the weak and missing
ones to their skill. Give a short, prioritized plan — not a wall of warnings.

### Step 5 — Re-audit before launch

Run it again as a go/no-go once the top gaps are closed.

## Examples

### Example 1: A marketing site built with v0 (no backend, no DB, no login)
**Action**: Profile says static/client-only. Most areas are N/A.
**Output**:
```
Profile: static site, no backend, no auth, no payments.
Applies:  app-security (check for exposed keys in the bundle), frontend-mobile-quality (responsive +
          a11y + performance), deployment-cicd (basic), production-readiness (the boring polish).
N/A:      auth, data, scaling, AI, API boundary, reliability, compliance, cost, monetization.
Do first: confirm no secrets are shipped in the client bundle  → app-security.
```

### Example 2: A B2B SaaS with login, Postgres, and a GPT feature, launching to ~500 users
**Action**: Profile triggers most areas; triage by blast radius.
**Output**:
```
Top gaps, by risk:
1. Security — are tables protected and keys server-side?  → app-security
2. Auth — can one tenant read another's data? test it.     → auth-access
3. AI cost — cap and cache the GPT calls before launch.     → ai-engineering
4. Observability — you have no error tracking.              → observability
Then: scaling (pooling/caching for 500 users), data (tenant isolation), deployment safety.
N/A right now: app-store compliance (web only), mobile deep links.
```

## Do / Don't

- **Do** profile the app before checking anything — relevance first.
- **Do** mark non-applicable areas as N/A and move on; don't manufacture work.
- **Do** rank by what actually hurts this app if shipped as-is.
- **Don't** treat every app as a multi-tenant, high-scale, LLM-powered, paid product.
- **Don't** call it "ready" until the top risk areas are addressed and you've re-audited.
